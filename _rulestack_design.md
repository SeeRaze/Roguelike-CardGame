## 0. Зачем (одним абзацем)

Игра про **авторство над собственным рулсетом**: игрок — процесс, редактирующий
запущенную программу. Чтобы «бесконечное безумие» не превратилось в неотлаживаемую
кашу из `if/else`, всё «ломающее правило» становится **данными в одном движке с
приоритетом и предохранителем** — `RuleStack`. Это не новая система с нуля: это
достройка паттерна, который в проекте уже применён ЧЕТЫРЕЖДЫ (`StatusRegistry`,
`ComboRegistry`, `DetonationRegistry`, `ForgeRegistry`), упорядоченного `ReactionOrder`
и защищённого `TriggerGuard`.

## 1. Решение по прогрессии (утверждено)

- Базовый забег стартует со стабильным `RuleStack` **пустым или с одними «Ставками»**
  (см. §6). Точки консультации стека вшиваем В ДВИЖОК С ПЕРВОГО ФРЕЙМА, даже когда
  стек пуст («базовая игра = пустой стек»).
- **Парадокс-режим** (автоматический `push` хаотичных правил) — отдельный **скрытый
  режим**, открывается по условию (пример: победа над боссом эт.100). Не ретрофитим в
  основной цикл — он эмерджит поверх готового движка.

## 2. Архитектура

### 2.1 `RuleMod` — атом слома (данные)
```
RuleMod:
  id:          str            # стабильный идентификатор
  name:        str            # для игрока
  description: str
  scope:       Scope          # КОГДА срабатывает (см. 2.2)
  priority:    int            # порядок ВНУТРИ scope (идея ReactionOrder)
  source:      str            # "stake" | "paradox" | "relic" | "committed"
  cost:        int = 0        # Энтропия, которую держит активной (0 для Ставок)
  predicate:   fn|None        # опц. условие активности (как requires в детонациях)
  apply(ctx):  ...            # хук: мутирует ctx/возвращает модификатор
```
Принцип как у реестров: **запись = эффект с приоритетом**, без хардкода в ядре.

### 2.2 `Scope` — НЕДОСТАЮЩАЯ ОСЬ (на каком слое вмешивается правило)
Обобщение `ReactionOrder` с боя на весь забег:
```
Scope:
  DECKBUILD     # старт забега — меняет стартовую колоду/правила
  RUN           # глобально на весь забег (множители, капы)
  ROOM_ENTER    # вход в узел карты
  COMBAT_START  # начало боя
  TURN_START / TURN_END
  REACTION      # внутри боевого конвейера реакций (стыкуется с ReactionOrder)
  DAMAGE        # расчёт урона (стыкуется с on_damage_calculated)
  ON_DEATH      # смерть игрока/врага → правки win-condition (Уроборос)
```

### 2.3 `RuleStack` — носитель состояния
- Живёт на `GameManager` (один на забег). Сериализуется в `SaveManager`
  (для «закоммиченных» правил — §8).
- `mods_for(scope) -> list[RuleMod]` — отфильтровать + упорядочить по priority.
- `apply(scope, ctx)` — единая точка: движок зовёт её в местах врезки (§2.4).
- `push(mod) / pop(id)` + проверка бюджета Энтропии (§5).

### 2.4 Точки врезки (где движок консультирует стек)
Вшиваем СРАЗУ (даже при пустом стеке — no-op):
| Scope | Место в коде |
|---|---|
| DECKBUILD | `GameManager` старт забега / выдача стартовой колоды |
| RUN | глобальные геттеры (множители урона/цен/дропа) |
| ROOM_ENTER | переход по узлу карты |
| COMBAT_START | `CombatManager.__init__` |
| TURN_START/END | `start_turn_phase` / `end_turn_phase` (рядом с relic-хуками) |
| REACTION | боевой конвейер (рядом с `_guarded`) |
| DAMAGE | `EffectCalculator.calculate_damage` (рядом с `on_damage_calculated`) |
| ON_DEATH | `_check_enemy_death` / `check_player_defeat` |

**Сосуществование с реликвиями:** реликвии НЕ мигрируем в RuleMods (работают, классы
переписывают — не трогаем). RuleStack — параллельный источник; реликвия может
`push`ить RuleMod, но система реликвий остаётся. Унифицируем только если докажет
ценность (🔵 развилка №2).

## 3. Три столпа движка
1. **Order/Scope** — `ReactionOrder` обобщён: когда и в каком порядке (priority внутри scope).
2. **Guard/Бюджет** — `TriggerGuard` обобщён: глубина каскада + **Энтропия** (§5) +
   Стабильность (амплитудный кап).
3. **Stack** — что активно прямо сейчас (новое).

## 4. Читаемость = архитектура
`RuleStack` — это И UI. Каждый активный RuleMod = видимая ховер-карточка в постоянном
списке (в порядке priority). Игрок ЧИТАЕТ список переписанных правил. Решает «кашу на
экране», учит системе, делает стек осязаемым. (Стыкуется с бэклогом HUD-иконок.)

## 5. Гард-рейлы (ОСНОВНАЯ инженерная работа, не эффекты)
- **Энтропия — валюта слома.** Держать RuleMod активным стоит Энтропии; бюджет
  ограничивает безумие ЭКОНОМИКОЙ, а не хардкод-капом. Четвёртая валюта рядом с
  золото=широта / FP=глубина / HP=риск.
- **Стабильность / Динамический Кап.** Мета-гард на ЧИСТЫЙ эффект: сила врага «из
  будущего акта» не выходит за «выигрываемый» конверт (HP в разумных рамках, а
  модификатор — из будущего тира). Аналог TriggerGuard, но для амплитуды баланса.
- **Детерминизм / скилл-инвариант.** RNG — только в МЕНЮ предложенных правок; КОММИТ —
  детерминированный, игроком. Хаос авторский, не сданный с руки.
- **Sim-native (козырь).** RuleMod — данные → бот тоже «надевает» их → дуал-сим/baseline
  автоматически меряет «этот сломанный рулсет выживаем?». Движок слома и движок
  баланса — ОДИН слой данных. Без этого самоизменяемый рулсет вручную не сбалансировать.

## 6. ВЕРТИКАЛЬНЫЙ СРЕЗ — Ставка/Анте (доказательство архитектуры)
Минимальная поверхность, проходящая весь путь данные→стек→движок→математика→сим.

**Ставка = RuleMod** (`source="stake"`, `cost=0`): пара **ограничение + глобальная
награда-множитель**. Примеры:
- «Аскет» — `scope=DECKBUILD`: стартовая колода ≤ 10 карт → `RUN` ×1.5 к урону.
- «Хрупкость» — `scope=RUN`: −50% макс. HP → ×2 к урону.
- «Безоружный» — `scope=DECKBUILD`: запрет карт защиты → +X к чему-то.

**Объём среза (этапы, 1 коммит = 1 шаг):**
1. `core/rules/RuleStack.py` — `Scope`(enum) + `RuleMod`(база) + `RuleStack`(push/pop/
   `mods_for`/`apply`). Чистый модуль, без pygame. + тесты.
2. Врезка ОДНОГО scope с наименьшим риском: **RUN-множитель урона** в
   `EffectCalculator` (там уже живёт паттерн `on_damage_calculated`). Инертно при пустом
   стеке. + тест.
3. 1–2 конкретные Ставки как RuleMod + DECKBUILD-ограничение (≤N карт). + тесты.
4. Активация Ставки в раннере симулятора → дуал-сим меряет «Ставка ВКЛ vs ВЫКЛ»
   (как forge/economy: параметр `stakes=None`, регресс-нейтрально по умолчанию).
   baseline-гард зелёный.
5. Минимальная активация в живой игре (выбор Ставки на старте — debug/простой экран;
   полный «скрытый режим» позже).

**Почему RUN/DAMAGE первым:** самая дешёвая врезка, которая прогоняет ВЕСЬ путь
(данные→стек→движок→урон→сим) на минимальной поверхности. Ограничение (≤N карт) —
на DECKBUILD. Доказали путь — дальше scope добавляются дёшево.

## 7. Долговая экономика (Долговой движок)
**Примитив овердрафта** (обобщение минус-HP Берсерка): ресурс может уйти В МИНУС в
пределах хода. Поддерживаемые: HP, энергия, золото, FP.
- Карта/способность стоит больше, чем есть → уходишь в минус (овердрафт).
- Долг = |минус|; пока в долгу, **долг → множитель** (глубже яма = больше выплата).
- **Гашение (settlement)** в конце боя/хода: выберись из долга или плати штраф
  (смерть/сжигание карты/потеря стата). «1 ход в коме на добив» Берсерка = окно
  гашения.
- Суть: **сила сейчас, платёж потом** — это заём времени, чистая «вторая экспонента»,
  риск-driven, не card-driven.

**Почему примитив, а не фича класса:** один механизм, десяток вкусов — Берсерк
овердрафтит HP, Шулер — золото («кровавый кредит»), Маг — энергию, Кузнец — FP.
Ставка «всё может уходить в минус» превращает весь забег в долговой.

**Математ. ручка:** `mult = 1 + debt/SCALE` (линейно) или `(1+r)^debt` (экспонента,
для потолка). Тюнится per-resource дуал-симом против кривой врага.

**Гард-рейл:** жёсткий пол долга ИЛИ суперлинейный штраф за непогашение (глубокий
долг = катастрофа, если не выплатил) → долг остаётся ВЫБОРОМ, не халявой.

**Флейвор (бренд):** игра ведёт твой долг как кредитор; пропустил гашение — статус
«коллектор» / «банкротство» в сейве. Связывает долг с метафорой кода/системы.

## 8. Концепт-бомба — «Патч-ноуты как прогрессия»
Перм. мета-прогресс = не анлоки, а **личный чейнджлог авторских правок правил**:
сломал в забеге → выжил у босса → можешь **закоммитить** правку в перм. рулсет
(ценой/трейдоффом). За забеги буквально пишешь свою версию игры. Лидерборд = «чей
рулсет самый сломанный-но-выживаемый». Технически = записи RuleStack, персистнутые в
`SaveManager`. Архитектура поддержит бесплатно. (Реализация — далеко; фиксируем как
северную звезду.)
